## 高质量输出的思维原则

### 核心转换：从"做了什么"到"发现/解决了什么"

> **警惕执行者思维**：执行者的逻辑是"我做了A、B、C"（过程驱动），高质量输出的逻辑是"问题是X，答案是Y，这意味着Z"（问题驱动）。无论是写代码、做分析还是写文档，都要问：**这是在回答一个问题，还是在描述一个步骤？**

**具体表现**：
| 执行者思维 ❌ | 问题驱动思维 ✓ |
|---|---|
| "我实现了函数X" | "这个函数解决了Y问题" |
| "我验证了模型" | "我发现了X，这揭示了Y" |
| 按时间/方法顺序组织 | 按问题递进组织 |
| 列举所有工作（First, Second, Third） | 整合核心洞见 |
| 显式预告下一步 | 逻辑自然过渡 |

### 叙事主线与逻辑衔接

> **一条主线贯穿始终**：任何输出都应该有一个清晰的故事线：问题是什么 → 为什么难/重要 → 核心洞见 → 证据/实现 → 意义/影响。每一部分都服务于这条主线。

**逻辑衔接的层次**：
1. **句子间**：上一句的结论自然引出下一句
2. **段落/模块间**：每个模块回答一个递进的问题
3. **全局**：开头提出的问题在中间被回答，在结尾被升华

**自检方法**：
- 删除所有显式过渡后，逻辑是否仍然连贯？
- 每个模块的开头是否点明它要解决的问题？
- 每个模块的结尾是否是take-home message（而非下一模块的预告）？

### 重点突出与信息层次

> **核心洞见要最显眼**：读者/用户的注意力是稀缺资源。核心内容应该在最显眼的位置，而不是被埋在细节中。

**信息分层原则**：
- **核心层**：只放理解主要论点所必需的信息
- **支撑层**：实现细节、方法说明
- **参考层**：完整推导、边界情况、robustness checks
- **元数据层**：配置、参数、版本信息

**措辞层次**：
- 核心发现/结论：assertive语气（"是"、"reveals"、"shows"）
- 支撑证据：neutral语气（"able"、"consistent with"）
- 局限/边界：honest语气（"constrained by"、"limited to"）

### 常见思维陷阱

1. **Defensive思维**："为了解决这个问题"、"我们尝试"——暗示你在被动响应，而非主动发现
2. **验证序列**：做X → 验证X → 再验证X——让人觉得你在证明自己对，而非推进理解
3. **清单式输出**：First, Second, Third——像汇报工作，不像提供洞见
4. **细节污染**：在核心层放实现细节——模糊重点，增加认知负担
5. **重复**：结尾重述开头——结尾应该是升华，不是总结

## 学术写作原则（针对高水平期刊）

### 核心思维转换：从技术报告到科学叙事

> **警惕技术报告思维**：技术报告的逻辑是"我们做了什么"（方法驱动），科学论文的逻辑是"我们发现了什么，这意味着什么"（问题驱动）。每写一段都要问：这是在回答一个科学问题，还是在描述一个技术步骤？

**具体表现与纠正**：
| 技术报告思维 | 科学叙事思维 |
|---|---|
| "We validate/confirm/test..." | "We find/show/identify..." |
| "模型预测X，我们验证了X" | "我们发现X，这揭示了Y" |
| Results按方法顺序排列 | Results按科学问题递进排列 |
| 每节末尾显式预告下一节 | 逻辑自然过渡，读者自己想知道下一步 |
| 列举所有贡献（First, Second, Third） | 用narrative statement整合核心发现 |

### 叙事主线与逻辑衔接

> **一条主线贯穿始终**：从Title到Conclusion，读者应该能感受到一个连贯的故事：提出问题 → 为什么难 → 我们的洞见 → 证据 → 意义。每一节都应该服务于这条主线，而不是独立的技术展示。

**逻辑衔接的三个层次**：
1. **段落间**：上一段的结论自然引出下一段的问题（不需要"We next..."）
2. **Section间**：每节回答一个递进的问题，读者读完自然想知道下一个问题的答案
3. **全文**：Introduction提出的问题在Results中被回答，在Discussion中被升华

**自检方法**：
- 删除所有显式过渡句后，逻辑是否仍然连贯？
- 每节的第一句话是否点明该节要回答的问题？
- 每节的最后一句话是否是该节的take-home message（而不是下一节的预告）？

### 重点突出与信息层次

> **核心发现要最显眼**：读者的注意力是稀缺资源。核心贡献应该在Abstract、Introduction末尾、Results的关键位置被明确陈述，而不是被埋在技术细节中。

**信息层次原则**：
- **正文**：只放理解核心论点所必需的信息
- **Methods**：说"怎么做"，不说"发现了什么"
- **SI**：技术细节、robustness checks、完整推导
- **Figure captions**：描述性，不重复正文的解释

**措辞层次**：
- 核心发现：assertive语气（"identifies", "reveals", "shows"）
- Robustness check：neutral语气（"confirms", "persists", "remains"）
- Limitations：honest语气（"constrained by", "limited to"）

### 常见陷阱

1. **Defensive writing**："To address this gap"、"A theoretical framework"、"We validate"——这些暗示你在填补空白或证明自己，而不是在做发现
2. **验证序列**：Theory → Validate → Validate → Validate——读者会觉得你在证明自己是对的，而不是在推进知识
3. **贡献清单**：First, Second, Third列举贡献——像PhD thesis答辩，不像期刊文章
4. **过度技术细节**：在正文中放代码路径、API参数、具体grid数值——这些属于SI
5. **重复**：Discussion重述Introduction的framing——Discussion应该是interpretation和implications，不是summary
